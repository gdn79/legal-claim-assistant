# Локальный ассистент для подготовки арбитражного иска

Проект принимает документы по делу, извлекает из них текст, отправляет данные в OpenAI-совместимый API, определяет стороны, сумму требований, достаточность документов, подсудность, госпошлину и формирует проект искового заявления в `.docx`.

Поддерживаемые форматы:

- PDF, включая сканированные PDF через локальный Tesseract OCR;
- DOCX;
- XLSX.

XLSX читается напрямую через ZIP/XML стандартной библиотекой Python, без `openpyxl` и `numpy`.

Старые форматы `.doc` и `.xls` напрямую не читаются: их нужно конвертировать в `.docx`, `.xlsx` или PDF.

```env

PROMPTS_FILE=data/prompts.json
RUSPROFILE_ENABLED=true
RUSPROFILE_TIMEOUT_SECONDS=15
```

Для Artemox используйте базовый URL `https://api.artemox.com/v1`. Клиент отправляет запросы через совместимый endpoint `/v1/chat/completions`.

Если используется обычный OpenAI API, `OPENAI_BASE_URL` можно оставить пустым или удалить.

`RUSPROFILE_ENABLED=true` включает получение недостающих реквизитов компаний по ИНН через `https://www.rusprofile.ru/`. Если сайт недоступен, заблокировал запрос или не вернул карточку, пайплайн не падает: остаются данные из документов и ИИ-анализа.

## OCR

OCR подключен через прямой запуск `tesseract.exe`, без `pytesseract`, `Pillow` и `numpy`. Это снижает риск проблем с Anaconda.

Для Windows обычно нужен путь:

```env
TESSERACT_CMD=C:\Program Files\Tesseract-OCR\tesseract.exe
```

Для русского языка в Tesseract должен быть установлен пакет `rus.traineddata`. Проверить можно так:

```powershell
& 'C:\Program Files\Tesseract-OCR\tesseract.exe' --list-langs
```

В списке должны быть `rus` и `eng`.


В локальном интерфейсе можно:

- загрузить PDF/DOCX/XLSX-документы;
- запустить OCR и анализ;
- видеть рабочий процесс обработки в боковой шкале: загрузка файлов, извлечение текста/OCR, запросы к ИИ, подготовка DOCX и этап оценки;
- остановиться после анализа на контрольной проверке 65%;
- вручную исправить истца, ответчика, договор, суммы, комплектность и регион ответчика;
- сформировать DOCX только после подтверждения проверенных данных;
- увидеть краткую сводку;
- скачать готовый `claim_draft.docx`;
- сохранить комментарий юриста для будущего обучения;
- перейти во вкладку редактирования промптов.

Если у стороны есть ИНН, программа может дополнить недостающие реквизиты через Rusprofile: ОГРН, КПП, юридический адрес, руководителя и ссылку на источник. Данные из документов не затираются, кроме явно некорректного ОГРН: если OCR записал в поле ОГРН почтовый индекс или другой номер не длиной 13/15 цифр, значение заменяется ОГРН из Rusprofile. Результат сохраняется в `analysis.json` в блоке `company_lookup`.

Нижняя фиксированная панель действий удалена из браузерного интерфейса. Основные действия выполняются внутри текущего экрана: выбор файлов, запуск анализа, подтверждение данных и скачивание DOCX.

Обработка в интерфейсе запускается асинхронно через `/cases/jobs`: браузер получает `job_id`, затем опрашивает `/cases/jobs/<job_id>` и обновляет этапы процесса без статичной заглушки. Обычный POST `/cases` оставлен как fallback, если JavaScript отключен.

Генерация DOCX после подтверждения также запускается асинхронно через `/cases/<case-id>/generate/jobs`. Кнопка подтверждения блокируется на время работы, чтобы не отправлять несколько одинаковых запросов к ИИ подряд.

Пайплайн теперь разделен на два обязательных этапа:

1. `analyze` — загрузка документов, OCR, извлечение текста, ИИ-анализ, расчет предварительной госпошлины и переход на экран проверки. На этом этапе готовность считается около 65%.
2. `generate_claim` — генерация DOCX только после ручного подтверждения или исправления анализа пользователем. После этого готовность считается около 95%.

Это сделано специально для качества: программа не должна молча формировать иск, если ИИ перепутал истца и ответчика, неверно прочитал сумму или не увидел недостающий документ.

После ИИ-анализа применяется deterministic-проверка акта сверки. Если в извлеченном тексте найдено итоговое сальдо вида `задолженность в пользу ... сумма`, эта сторона считается кредитором и ставится истцом. Это правило имеет приоритет над ролями из договора (`Заказчик`/`Исполнитель`), потому что в споре о взыскании важен не общий статус стороны в договоре, а финальное сальдо взаиморасчетов. Если акт сверки указывает кредитора, совпадающего с ответчиком, программа автоматически меняет истца и ответчика местами и записывает причину в `party_reasoning.reconciliation_balance`.

После определения сторон программа выбирает `claim_scenario` — модель будущего иска. Поддерживаются сценарии: `contractor_payment` (исполнитель взыскивает оплату), `reconciliation_balance` (задолженность по акту сверки), `customer_refund` (заказчик взыскивает переплату/возврат), `generic_debt` (общая задолженность). Модель показывается на экране проверки и может быть исправлена пользователем до генерации DOCX.

`claim_scenario` передается в промпт генерации. Для акта сверки и возврата переплаты модель запрещает писать старую фабулу “ответчик не оплатил выполненные истцом работы”, если истец является заказчиком или кредитором по сальдо. После генерации текст дополнительно проверяется на такие противоречия, предупреждения сохраняются в `review.json`.

Перед записью в DOCX текст иска проходит очистку от разговорных артефактов ИИ: markdown-разделителей `---`, блоков ```, хвостовых фраз вроде `Если хотите...`, `Если необходимо...`, `При необходимости...`, `Хотите, чтобы я...`, `Я могу подготовить...`, `Готов подготовить...`, а также финальных блоков `Примечания`, `Важные уточнения`, `Дополнительные уточнения`. Такой текст не попадает ни в Word-файл, ни в training snapshot.

Также DOCX-builder удаляет явные технические заголовки разделов, если модель всё же вставила их в текст: `Истец`, `Ответчик`, `Шапка`, `Суть спора`, `Цена иска`, `Сумма иска`, `Госпошлина`, `Обстоятельства дела`, `Правовое обоснование`, `Правовые основания`, `Расчет`, `Претензионный порядок`, `Просьба истца`, `Просительная часть`, `Приложения`, `Дата подачи иска`, `Представитель истца`. Markdown-выделение `**...**` внутри строк тоже очищается. Итоговый Word должен быть цельным исковым заявлением с обычными абзацами, без видимой разбивки на такие группы.

Формат Word-файла: основной текст Times New Roman 14 pt, абзацы выровнены по ширине страницы, первая строка абзаца имеет отступ. Поля страницы: сверху 2 см, снизу 2 см, слева 3 см, справа 1 см. Лишние подряд идущие пустые строки сжимаются до одного пустого абзаца.

В генерацию передается не только сумма госпошлины, но и текст формулы расчета. Например для цены иска 275 000 руб.: `10 000 руб. + 5% от суммы, превышающей 100 000 руб. = 18 750 руб.` Это предотвращает ошибочную формулировку “3% от суммы иска”.

Настройки из `.env` перечитываются веб-интерфейсом на каждый запрос. Если вы изменили `OPENAI_API_KEY`, `OPENAI_MODEL` или timeout, обычно достаточно обновить страницу; при сомнениях перезапустите сервер.

## Редактирование промптов

Промпты по умолчанию находятся в файле:

```text
src/legal_claim_assistant/prompts.py
```

В веб-интерфейсе есть отдельная вкладка:

```text
http://127.0.0.1:8000/prompts
```

Там можно редактировать четыре блока:

- `analysis_system` — системный промпт юридического анализа;
- `analysis_user` — пользовательский промпт анализа документов;
- `claim_system` — системный промпт подготовки иска;
- `claim_user` — пользовательский промпт текста иска.

После сохранения промпты записываются в:

```text
data/prompts.json
```

Следующие запуски анализа используют `data/prompts.json`. Если файл удалить или нажать сброс в интерфейсе, система вернётся к значениям из `src/legal_claim_assistant/prompts.py`.

Важно: в промптах должны остаться служебные маркеры:

- в `analysis_user`: `{documents}`;
- в `claim_user`: `{analysis}`, `{court}`, `{state_duty_rub}`.

При сохранении редактор проверяет эти маркеры и JSON-шаблон в `analysis_user`, если в промпте есть блок `Формат JSON:`. Если в примере JSON пропущена запятая или сломаны скобки, промпт не сохраняется и интерфейс показывает ошибку.

Текущий пользовательский промпт хранится в `data/prompts.json`. Если файл существует, API использует его, а не короткие значения из `prompts.py`. Короткие значения применяются только как fallback, когда пользовательский файл отсутствует, поврежден или конкретное поле пустое.

## CLI-режим

```powershell
python -m legal_claim_assistant run .\docs\dogovor.pdf .\docs\akt.docx --case-name "Долг по услугам"
```

Или после установки проекта:

```powershell
legal-claim-assistant run .\docs\dogovor.pdf .\docs\akt.docx --case-name "Долг по услугам"
```

Результаты сохраняются в:

```text
data/cases/<case-id>/output
```

Основные файлы результата:

- `texts/*.txt` — извлечённый текст;
- `analysis.json` — структурированный юридический анализ;
- `analysis.reviewed.json` — исправленный пользователем анализ, если данные правились перед генерацией;
- `review.json` — статус проверки, процент готовности, предварительная госпошлина и недостающие документы;
- `case_summary.json` — краткий итог;
- `claim_draft.docx` — проект искового заявления.

Копия данных для доработки промптов и обучения сохраняется в:

```text
data/training
```

## Частые проблемы

Если кнопка скачивания выдаёт 500, проверьте `logs/assistant.log` и наличие:

```text
data/cases/<case-id>/output/claim_draft.docx
```

Если API возвращает ошибку по модели, проверьте:

```env
OPENAI_MODEL=...
OPENAI_BASE_URL=https://api.artemox.com/v1
```

Если OCR не работает, проверьте:

```powershell
& 'C:\Program Files\Tesseract-OCR\tesseract.exe' --version
& 'C:\Program Files\Tesseract-OCR\tesseract.exe' --list-langs
```

Если запуск идёт из Anaconda `(base)` и возникают странные ошибки пакетов, лучше использовать отдельную `.venv`.

Если интерфейс пишет `нужен OPENAI_API_KEY`, а ключ есть в `.env`, проверьте, откуда запущен сервер. Для запуска из текущего кода используйте:

```powershell
cd D:\CODEX\Assistent
$env:PYTHONPATH="D:\CODEX\Assistent\src"
D:\Anaconda\python.exe -m legal_claim_assistant serve
```

Если файл с русским именем сохраняется без расширения, приложение автоматически возвращает исходное расширение `.pdf`, `.docx` или `.xlsx`. Это нужно для имён вроде `Договор.pdf`, которые `secure_filename` может превратить в `pdf`.

## Изменения 2026-05-09: режим готовности и понятный браузерный сценарий

Программа переведена на более строгую схему подготовки иска: сначала собирается карточка дела, затем пользователь проверяет ключевые факты, после этого формируется DOCX и запускается автоматический контроль текста.

Что добавлено:

- модуль `src/legal_claim_assistant/readiness.py`, который строит карточку дела и считает процент готовности по чек-листу;
- проверка готовности больше не является статичной: учитываются документы, стороны, модель иска, цена иска, претензионный порядок, реквизиты компаний, приложения и итоговый текст DOCX;
- после генерации иск проверяется на критические ошибки: перепутанные роли сторон, неправильная фабула для акта сверки/возврата, разговорные хвосты ИИ, markdown-разметка;
- если автоматический контроль нашел критические ошибки, скачивание DOCX блокируется до исправления карточки дела или промпта;
- основной экран переписан как понятный рабочий маршрут: загрузка документов, проверка карточки, исправление рисков, формирование и скачивание DOCX;
- в интерфейсе появился реальный чек-лист готовности с оценкой каждого блока;
- промпт генерации усилен: ИИ запрещено менять роли сторон, называть платежное поручение актом, придумывать отсутствующие документы, задавать вопросы пользователю и добавлять фразы вроде `если хотите` или `могу подготовить`.

Логика готовности:

- документы: договор и доказательства исполнения/оплаты;
- стороны: наименование, ИНН, ОГРН, адрес истца и ответчика;
- модель иска: долг за работы, акт сверки, возврат аванса/переплаты или общий долг;
- суммы: основной долг, проценты, неустойка и итоговая цена иска;
- претензионный порядок: претензия и доказательства отправки;
- реквизиты компаний: КПП, руководитель/представитель, актуальные данные;
- приложения: документы, которые должны идти вместе с иском;
- текст иска: итоговая проверка DOCX на противоречия и артефакты ИИ.

Важно: 100% в программе означает техническую готовность по чек-листу, а не гарантию судебного результата. Перед подачей документ все равно должен быть проверен юристом.

По состоянию на проверку 2026-05-09 страница Банка России показывала обновление от 2026-05-07 и ключевую ставку 14,50%.

Правило расчета:

- проценты 395 ГК РФ считаются по формуле `сумма долга × ключевая ставка / 100 / 365 × количество дней`;
- период считается включительно по датам, введенным пользователем;
- если за период ставка менялась, расчет разбивается на интервалы;
- договорная неустойка считается по формуле `сумма × процент в день / 100 × количество дней`;
- если даты или ставка не заполнены, программа не придумывает расчет и оставляет предупреждение.

Ограничение: расчетчик не заменяет юридическую проверку. Для подачи иска нужно проверить дату начала просрочки, дату окончания расчета, применимость договорной неустойки, возможность снижения неустойки по ст. 333 ГК РФ и актуальность ключевых ставок ЦБ РФ.

- при невозможности записи в `.tmp/ocr` пользователь получает понятное сообщение с путем к проблемной папке.

## Изменения 2026-05-09: правовая валидация и self-critique second pass

После генерации текста все ссылки вида `ст. X ГК РФ`, `ст. Y АПК РФ`, `ст. Z НК РФ` сверяются со справочниками реально существующих статей:

- АПК РФ: статьи 1–342
- НК РФ: статьи 3–460 с подстатьями (333.21, 333.33 и др.)

Если найдена ссылка на несуществующую статью, готовность иска блокируется. Справочники находятся в `readiness.py`.

### Проверка обязательных реквизитов иска (ст. 125 АПК РФ)
После генерации проверяется наличие:
- наименование арбитражного суда;
- данные истца и ответчика;
- цена иска;
- просительная часть (Прошу);
- перечень приложений;
- дата;
- упоминание госпошлины.

Отсутствие любого элемента добавляется в warnings readiness-отчёта.

### Self-critique second pass
После генерации текста иска делается второй AI-запрос (отдельная модель в той же сессии) для проверки по чеклисту:
- соответствует ли текст выбранному сценарию;
- корректны ли ссылки на статьи;
- указаны ли суд, дата, подпись;
- нет ли разговорных фраз ("если хотите", "могу подготовить");
- не упоминаются ли документы, которых нет в материалах дела.

Замечания self-critique добавляются в `validation_warnings`. Если self-critique временно недоступен, генерация не падает — пропускается с предупреждением в лог.
